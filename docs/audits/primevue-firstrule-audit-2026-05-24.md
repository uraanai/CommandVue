# PrimeVue-first compliance audit — 2026-05-24

Generated for Prompt 2, Phase 2.2 after the user accepted **Option C — hybrid** in ADR 0002. Records every UI element in `src/components/` against the accepted rule and produces the migration plan that Phase 2.3 will execute.

**The recommendations below follow ADR 0002's split:**

- Density-critical or project-API-masked primitives → wrapper in `src/components/ui/<Name>.vue`.
- General-purpose primitives → install via `npx volt-vue add <Name>` to `src/volt/<Name>.vue`.
- Raw HTML interactive elements (`<button>`, `<input>`, `<select>`, `<textarea>`) → migrate to the appropriate wrapper or Volt component.
- `primevue/datatable` consumers → out of scope (governed by ADR 0001; tracked separately in `docs/audits/datatable-usage-inventory.md`).

## Unclear items requiring user input

These three are flagged `unclear` in the table above. Phase 2.3 should not start them until resolved.

1. **`ChromeSlot.vue` — the "+ Add item" dropdown (lines 84–110).** The whole construct (trigger button + outside-click popover + item list) is a hand-rolled dropdown. Two options:
   - **(1a)** Convert the whole construct to a single PrimeVue `Menu` (or Volt `Menu`) usage. The trigger becomes an `<IconButton>`; the popover items become `MenuItem`s. Cleanest and matches the rule.
   - **(1b)** Convert only the trigger and item buttons individually to `<IconButton>` / `<Button>`; leave the popover hand-rolled.
   - **Recommendation:** **(1a)**. Slightly larger refactor but eliminates a custom popover layer.
   - **Question:** Go with (1a)? If yes, Phase 2.3's chrome PR includes this conversion.

2. **`CommandPalette.vue` — result-list items (line 189).** The list buttons are virtualization-style row affordances inside a custom Spotlight UI. Two options:
   - **(2a)** Migrate each list button to `<Button variant="ghost">`. Wrapper overhead per row.
   - **(2b)** Keep raw `<button>` with an exception note in the file header. Each row is essentially a presentational element, not a free-floating button.
   - **Recommendation:** **(2b)** — document the exception. The close button (line 170) and search input (line 162) DO migrate; the result rows stay raw with a header comment.
   - **Question:** Accept (2b) as the documented exception? Or insist on (2a) for consistency?

3. **`ComponentsPanel.vue` — panel-type tile grid (line 104).** Each tile is a two-line button with icon + title + description. Two options:
   - **(3a)** Migrate each tile to `<Button>` with a custom slot for the two-line content. Many `<Button>`s in a grid.
   - **(3b)** Migrate the whole panel to PrimeVue `DataView` (`layout="grid"`) which models category-grouped item grids natively. Each tile becomes a `DataView` item template; selection is the consumer's responsibility.
   - **Recommendation:** **(3b)**. PrimeVue ships exactly the right component; the current hand-rolled grid is a clear case where library-first applies.
   - **Question:** Go with (3b)? If yes, Phase 2.3's panels PR includes this rebuild.
