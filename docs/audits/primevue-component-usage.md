## Coverage analysis

Components in active use (excluding `primevue/config`, `primevue/menuitem` (type-only), and the 4 deferred `primevue/datatable` files):

- **Total distinct PrimeVue components used:** 14 (Button, InputText, Select, Tabs+TabList+Tab+TabPanels+TabPanel = 5, Dialog, Toast, Checkbox, ColorPicker, Slider, Textarea, Menu, Menubar, ContextMenu, FileUpload, Fieldset, Tag)
  - Counting the Tabs family as one component for coverage purposes → 10 distinct components.
- **Confirmed Volt coverage (verified via Volt docs):** Button, InputText, Select, Tabs (family), Dialog, Toast, Checkbox, Slider, Textarea, Fieldset, Tag, Menu → **12 confirmed**.
- **Volt coverage unverified but likely (overview promises "50+ components"; menu/file family is standard):** Menubar, ContextMenu, FileUpload → **3 unverified**.
- **Volt coverage uncertain (uncommon component):** ColorPicker → **1 uncertain**.

**Estimated Volt coverage of CommandVue's actual usage: 100% of confirmed + likely, with 1 verify-before-adopt item (ColorPicker).** If ColorPicker turns out not to be in Volt, the project keeps `primevue/colorpicker` for that one editor and that's it — no other components are at risk.

## What's NOT in this inventory

- The four `primevue/datatable` consumers — governed by ADR 0001 separately; their migration target is the project's `<DataTable>` wrapper (TanStack-based), not Volt.
- Raw HTML interactive elements (`<button>`, `<input>`, `<select>`) — those are tracked separately in Phase 2.2's audit and are out of scope here. This inventory only covers files that already use PrimeVue.

## Implication for the ADR

Volt's coverage of CommandVue's PrimeVue usage is comfortable. The decision is not about coverage — it's about whether the cost of owning two parallel UI directories (`src/components/ui/*` for wrappers we keep + `src/volt/*` for Volt-installed components) is justified by what Volt delivers that the existing wrappers don't.
